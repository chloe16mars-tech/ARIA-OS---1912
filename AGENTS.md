# AGENTS.md — ARIA-OS engineering rules

> Single source of truth for AI agents and human contributors working on this repo.
> The README covers *what* the project is and *how* to run it; this file covers *how to write code* in it.

**Stack:** Angular 21 (standalone, signals, zoneless) · Express 5 (Angular SSR + REST) · Supabase (Postgres + Auth + Realtime) · Google Gemini · Capacitor 8 · Tailwind CSS 4 · TypeScript 5.9 (strict).

---

## Objective

Ship correct, simple, maintainable, performant code. In that order.

If a rule below conflicts with a better-justified design choice, write the rationale in the PR description and proceed. Mechanical compliance is not the goal.

---

## Core rules

1. **Read before you write.** The relevant files, the AGENTS.md, the existing patterns. Match the conventions you find.
2. **Smallest diff that solves the problem.** No drive-by refactors. No speculative abstractions. No "while we're here" rewrites.
3. **No `any`.** Use proper types, generics or `unknown` + narrowing. The TypeScript config is `strict`.
4. **No dead code.** Delete it; don't comment it out. Git remembers.
5. **No hardcoded secrets, hosts or model IDs.** Everything tunable lives in `.env` and `src/environments/*` (the latter generated by `scripts/set-env.mjs`).
6. **DRY, but only when duplication is real.** Three near-copies are fine; six are a refactor.

## Architecture

### Frontend (`src/app/`)

- **Standalone components only.** No NgModules.
- **Signals for reactive state** — `signal()`, `computed()`, `effect()`. Avoid `BehaviorSubject` for component state.
- **`inject()`** instead of constructor parameters.
- **OnPush change detection** on every component (the app runs zoneless — anything else is dead weight).
- **One component per file.** Inline templates are fine up to ~80 lines; beyond that, split or extract a child component.
- **HTTP via `provideHttpClient`** when possible. `fetch()` is acceptable when streaming SSE or when the consumer is a plain async function.

### Backend (`src/server/`)

- **Pure helpers** (prompt building, URL guards) live in their own files and are unit-tested with Vitest.
- **Routes are thin.** They validate input, call helpers, and serialise the response — nothing else.
- **Every protected endpoint** verifies the JWT via `supabaseAdmin.auth.getUser(jwt)`.
- **Every input** is validated with Zod before it reaches the business logic.
- **Rate limiting** is mandatory for routes that hit Gemini or mutate auth state.

### Database (`supabase/migrations/`)

- **All schema, RLS and RPC** changes go through a versioned migration. Never edit the database from the dashboard without committing the SQL.
- **RLS on by default** for every table that holds user data.
- **RPCs are `security definer`** and explicitly granted to `authenticated` / `service_role`.

---

## Security baseline (do not regress)

- The service role key and the Gemini key live server-side. Never import them from `src/app/`.
- The frontend bundle only sees variables prefixed `PUBLIC_*`.
- `bypassSecurityTrust*` calls are gated by an explicit allow-list (see `src/app/pipes/safe-url.pipe.ts`).
- HTML rendered from model output is entity-escaped *before* any decorative tag is added (see `src/app/pipes/script-format.pipe.ts`).
- User-supplied URLs go through `isSafePublicUrl` before any server-side fetch or before being placed in an iframe.
- `trust proxy` is honored so that `express-rate-limit` keys on the real client IP.

## Testing

- **Pure Node helpers** → Vitest (`src/server/**/*.spec.ts`, `npm run test:server`).
- **Angular components / pipes** → `ng test` (`src/app/**/*.spec.ts`).
- A bug fix without a regression test gets bounced.

---

## Commits & PRs

- Conventional commits encouraged (`feat:`, `fix:`, `chore:`, …) but not enforced.
- One logical change per commit. Format/lint commits are kept separate.
- A PR description explains *why* — the diff already explains *what*.

---

## What to avoid

| ❌ | ✅ |
|---|---|
| `as any` | proper types or `unknown` + guards |
| Hardcoded URLs / model IDs | `process.env` or `environment.*` |
| `setInterval` polling for promises | a real `Promise` you await |
| `new Promise(async (res, rej) => …)` | a plain `async` function |
| Wildcard CORS in production | explicit `ALLOWED_ORIGINS` |
| `unsafe-eval` in prod CSP | hashed bundles + `'self'` |
| Raw SQL string interpolation | Supabase query builder / parameterised RPC |
| `console.log` in services | `LoggerService` (silent in prod) |

---

## Final rule

You are a senior engineer. Default to the boring, well-typed, well-tested solution. Justify everything else.
